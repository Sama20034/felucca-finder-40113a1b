

## المشكلة

لا، كلمة المرور **ليست** كلمة مرور Gmail. المشكلة إن Supabase لما بيعمل invite، بيسجّل دخول المستخدم تلقائياً عن طريق الـ token في الرابط. الكود الحالي مش بيقدر يكتشف إن ده invite لأن Supabase (في الإصدارات الجديدة مع PKCE flow) بيشيل الـ hash من الـ URL قبل ما الكومبوننت يقرأه، فالمستخدم بيتوجه للداشبورد مباشرة بدون ما يحط باسورد.

النتيجة: المستخدم المدعو **ما عندوش باسورد** — لو عمل sign out مش هيقدر يرجع يدخل تاني!

## الحل

### 1. تعديل `src/pages/Auth.tsx`
- بدل ما نعتمد على `window.location.hash` (غير موثوق)، نستخدم `onAuthStateChange` event
- لما الـ event يكون `PASSWORD_RECOVERY` → نعرض فورم تعيين كلمة المرور
- نضيف كمان check: لو المستخدم جديد (created_at قريب جداً) وما عندوش `last_sign_in_at` سابق → نعرض فورم تعيين كلمة المرور

### 2. تعديل Edge Function `invite-admin`
- نغير الـ invite عشان يستخدم `type=recovery` بدل invite عادي
- أو نضيف `data` metadata للمستخدم المدعو فيها `needs_password: true`

### 3. تعديل `src/contexts/AuthContext.tsx`
- نضيف flag `isInvitedUser` في الـ context
- لما `onAuthStateChange` يطلع `PASSWORD_RECOVERY` event، نمنع الـ redirect ونخلي صفحة Auth تعرض فورم الباسورد

### السيناريو بعد التعديل:
1. المدعو يضغط رابط الدعوة
2. Supabase يسجل دخوله تلقائياً
3. النظام يكتشف إنه مستخدم مدعو (أول مرة يدخل)
4. يعرضله فورم "اختار كلمة مرور" بدل ما يوديه الداشبورد
5. بعد ما يختار الباسورد، يدخل الداشبورد

### ملاحظة مهمة
المستخدم اللي اتعمله invite حالياً (Ahmed) محتاج يعمل password reset من صفحة تسجيل الدخول عشان يقدر يحط باسورد — أو تبعتله invite جديد بعد التعديل.
